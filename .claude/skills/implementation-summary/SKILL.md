# Implementation Summary

## When to use
요청된 마이그레이션, 버그 수정, 또는 기능 개발 작업이 모두 완료되고 마지막으로 사용자에게 결과 리포트를 제출할 때 사용합니다.

## Workflow
1. **변경 범위 파악**: 이번 작업으로 수정되거나 생성된 모든 파일 목록을 취합합니다.
2. **영향도 정리**: 각 구현체(JSP, SSR, CSR_Back, CSR_Front) 별로 나누어 핵심 변경 사항을 정리합니다.
3. **Contract 점검**: 프론트엔드/백엔드 간 API Contract 변경이 있었다면 명시합니다.
4. **테스트 및 검증 결과**: 코드에 대해 수행한 테스트나 검증 상태를 기록합니다.
5. **후속 제안**: 이후 필요한 개선 사항이나 이어서 작업해야 할 다음 단계를 제안합니다.

## Rules
- 단순 파일 변경 리스트 나열에 그치지 않고, "기능 동등성 확보 여부"와 "아키텍처 변화"를 중점적으로 설명하십시오.
- 구현체별로 섹션을 나누어 가독성 좋게 포맷팅해야 합니다.

## Output
- 구현체별 변경 사항 핵심 요약
- 전체 수정/생성 파일 목록
- API Contract 변경 여부 명시
- 테스트/검증 결과 보고
- 후속 작업(Next Steps) 제안
